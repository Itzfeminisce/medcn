# AGENTS.md — working on medcn

medcn is a shadcn-style registry + doc site for health/medical UI components. Read [PRD.md](./PRD.md) for product scope and [docs/DESIGN.md](./docs/DESIGN.md) for visual rules before changing anything user-visible.

## Commands

```bash
pnpm install            # slow on this machine (10–30 min); run in background
pnpm dev                # doc site on :3000 (the user usually runs this themselves)
pnpm registry:build     # regenerate public/r/*.json + public/llms.txt (gitignored)
pnpm typecheck
pnpm build              # registry:build + next build — the full gate
```

Gotchas: pnpm build-script approvals live in `pnpm-workspace.yaml` (`allowBuilds:`), not package.json. tsconfig has `noUncheckedIndexedAccess` — array indexing yields `T | undefined`.

## Architecture (do not change without asking)

- **`registry/medcn/<name>/` is the source of truth.** One folder per component: `<name>.tsx` + `<name>.demo.tsx` + `meta.json`. Everything else (registry JSON, llms.txt, doc pages, sidebar) is generated or derived from these folders.
- **Authoring imports** inside registry files: `@/registry/medcn/lib/utils` and `@/registry/medcn/<dep>/<dep>`. `scripts/build-registry.ts` rewrites them for consumers to `@/lib/utils` and `@/components/ui/<dep>`. Never hand-write consumer paths in registry source.
- **Registry output** is static JSON in `public/r/` (shadcn `registry-item.json` schema, source inlined in `files[].content`). Base URL comes from `REGISTRY_URL` (default `https://medcn.dev/r`). `registryDependencies` named in meta.json resolve to our URLs, except shadcn built-ins (`utils`) which pass through.
- **The doc site dogfoods the registry — strictly.** Site chrome must be composed from `registry/medcn` components (shadcn conventions) wherever one exists. If the site needs a new reusable primitive, add it to the registry first, then consume it. Site-only helpers (`components/`) are allowed only for doc-site mechanics (CodeBlock, CommandPill, SidebarNav, ThemeToggle, ComponentPreview) and must still follow shadcn conventions: `cn()` merge, `data-slot` where sensible, cva for variants, no forwardRef (React 19).
- **Code shown to users is always shiki-highlighted** via `components/code-block.tsx` (server component, `github-dark-default`, fixed dark surface in both themes). Never render a bare `<pre>` for user-facing code.

## Component conventions

Copied from shadcn/petals_ui practice: `React.ComponentProps<...>` prop types, cva variants exported alongside the component, `cn` on every className, `"use client"` only where hooks/Radix require it, custom `title: ReactNode` props must `Omit<..., "title">` from HTML props. No Petals branding (no brand-pink/brand-teal) — medcn is theme-neutral clinical.

## meta.json schema

```jsonc
{
  "name": "vitals-card",          // must equal the folder name
  "title": "Vitals Card",
  "description": "…",              // one sentence, ends with a period
  "category": "vitals",            // key of CATEGORIES in lib/registry.ts
  "type": "registry:ui",
  "version": "0.1.0",              // bump on any change to the .tsx
  "dependencies": ["lucide-react"],          // npm packages
  "registryDependencies": ["utils", "card"], // registry item names
  "links": { "radix": "https://…" },         // Radix primitive it wraps; use { "shadcn": "https://…" }
                                             // instead when there's no Radix but a shadcn counterpart;
                                             // omit for medcn-original health components
  "props": [{ "name", "type", "default?", "description?" }],
  "clinicalNotes": "…",            // REQUIRED for health components
  "notes": "…"                     // internal; never rendered on the site
}
```

## Adding a component (the loop)

1. Create `registry/medcn/<name>/` with component + demo + meta.json (authoring imports).
2. Register the demo in `components/demos.tsx`.
3. `pnpm registry:build && pnpm typecheck && pnpm build`.
4. Verify both themes (dark is default) and the Preview/Code tabs render.
5. Quality bar (PRD §7): honest demo, clinical notes for health components, no Petals brand variants, installs cleanly via `npx shadcn add` with its whole dependency chain.

## Content rules (from docs/DESIGN.md)

- No internal/meta copy on pages: no porting notes, roadmap promises, or positioning essays. Those live in README/PRD only.
- One neon accent (cyan-teal `--primary`); glow (`shadow-glow`) only on interactive/hero elements; the demo is the hero.
- Dark is the default theme; every change must look right in both themes.

## Agent-facing surfaces (keep them working)

- `public/llms.txt` — generated by `registry:build`; regenerate after adding/renaming components.
- Registry item `meta` carries `docs`, `props`, `clinicalNotes`, `links` — agents rely on these; don't strip them.
- `/docs/agents` documents the agent workflow (llms.txt, registry endpoints, shadcn MCP) — update it if endpoints change.
